## 1. Goals

- Extend the `instrument-reg resolve` CLI command to accept JSONL (JSON Lines) from stdin pipe, in addition to the existing single-JSON support added in the prior session.

## 2. Work Done

- Read `src/instrument_registry/cli/resolve.py` and `src/instrument_registry/cli/common.py` to understand current implementation.
- Reviewed prior session notes (`docs/sessions/2026-04-22T19-14.md`) for context on the single-JSON pipe work.
- Modified `src/instrument_registry/cli/resolve.py`:
  - `_read_pipe()` return type changed from `dict[str, Any]` to `list[dict[str, Any]]`. It now tries `json.loads()` first (handles single object or JSON array), then falls back to line-by-line JSONL parsing with per-line error messages (`Invalid JSON on line N`).
  - Resolution logic extracted to a new `_resolve_criteria()` helper that owns the full cycle: criteria build → `resolve_and_persist` → price verification → output.
  - `command()` stdin branch loops over all records returned by `_read_pipe()`, calling `_resolve_criteria()` per record.
  - Single-record stdin JSON and multi-record JSONL streams are both handled without duplication.
- Added 4 new tests to `tests/test_cli_coverage.py`:
  - `test_cli_resolve_single_json_pipe` — existing single-JSON behavior is preserved
  - `test_cli_resolve_jsonl_pipe_multiple_records` — two JSONL lines produce two `resolve_and_persist` calls
  - `test_cli_resolve_jsonl_pipe_skips_blank_lines` — blank lines in JSONL input are ignored
  - `test_cli_resolve_jsonl_pipe_invalid_line_fails` — malformed JSON on any line exits with code 1
- Ran full test suite: 28 passed, 0 failed.

## 3. Decisions Made

| Decision | Rationale |
|----------|-----------|
| `_read_pipe()` returns `list[dict]` always | Unifies single-record and multi-record paths so `command()` always loops, eliminating branching |
| Try `json.loads(raw)` before JSONL line splitting | Correctly handles pretty-printed multi-line JSON objects (which would fail line-by-line parsing) |
| Extracted `_resolve_criteria()` helper | Avoids duplicating the full resolve → verify → output logic between the query path and the pipe loop |
| JSON array input (`[{...}, {...}]`) also supported | Falls out naturally from the `json.loads` branch with no extra work |
| Fail fast on first invalid JSONL line | Consistent with project fail-fast rules; partial output would be confusing |

## 4. Decisions Refused / Alternatives Rejected

| Rejected Approach | Reason |
|-------------------|--------|
| Keeping `_read_pipe()` returning a single dict and looping inline | Would require duplicating the criteria-build and output logic; extraction is cleaner |
| Continuing on invalid JSONL lines and reporting errors at the end | Violates the project fail-fast rule |
