## 1. Goals

- Add stdin pipe support to the `resolve` CLI command so that a JSON instrument record can be piped directly: `echo '{...}' | uv run instrument-reg resolve`
- The piped JSON format matches: `{"isin":"...","symbol":"...","currency":"...","asset_class":null,...}`

## 3. Decisions Made

| Decision | Rationale |
|----------|-----------|
| Make `query` optional with `typer.Argument(None, ...)` instead of adding a separate flag | Follows Unix convention (`-` for stdin); agents can omit the positional and pipe instead |
| CLI flags (`--currency`, `--asset-class`, etc.) take precedence over piped JSON | Allows callers to override individual fields from the pipe without reconstructing the whole JSON |
| `_read_pipe()` uses `raise AssertionError("unreachable")` after `exit_with_error` calls | Satisfies mypy return-type checking since `exit_with_error` raises `SystemExit` but is not typed `NoReturn` |
| IBKR conid fast-path guarded with `query is not None` check | Prevents `isdigit()` TypeError on `None` when no query is provided |
| `query_label` derived from `isin or symbol or "(stdin)"` | Keeps error messages meaningful even when the query came from structured pipe input |

## 4. Decisions Refused / Alternatives Rejected

| Rejected Approach | Reason |
|-------------------|--------|
| Adding a dedicated `--from-stdin` flag | Unnecessary â€” Unix convention is to detect piped stdin via `sys.stdin.isatty()` |
| Reading stdin only when `query == "-"` | Less ergonomic for agents; the user's requested syntax uses no positional argument at all |
